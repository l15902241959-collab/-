# Day 1 推荐系统全局地图

## 1. 推荐系统三阶段/四阶段架构

工业推荐系统通常采用多阶段架构，核心原因是全量 item 数量巨大，无法直接对所有 item 使用复杂模型精排。因此系统一般按照“候选规模逐步缩小、模型复杂度逐步提高”的思路组织。

典型链路如下：

```text
全量 item
→ 召回
→ 粗排
→ 精排
→ 重排
→ 推荐结果
```

### 1.1 召回 Candidate Generation / Retrieval

召回阶段的目标是从千万级、亿级 item 中快速选出几百到几千个候选 item。

召回的核心要求是：

```text
高效率
高覆盖
尽量不遗漏潜在相关 item
```

召回阶段通常采用多路召回。不同召回通道可以刻画不同兴趣来源或业务目标，例如：

```text
协同过滤召回
内容相似召回
热门召回
向量召回
图召回
序列召回
冷启动召回
```

召回阶段常见方法包括：

```text
协同过滤
矩阵分解
FM / FFM
图模型
双塔模型
DNN 召回
Deep Retrieval
```

在向量召回中，模型通常学习：

```text
user embedding
item embedding
```

然后通过相似度检索得到候选 item：

```text
score(user, item) = similarity(user_embedding, item_embedding)
```

为了提升检索效率，线上通常不会对全量 item 暴力打分，而是使用近似最近邻检索 ANN，例如：

```text
Faiss
HNSW
ScaNN
Ball Tree
```

### 1.2 粗排 Pre-ranking

粗排阶段接收召回返回的候选 item，并进一步筛选出更小规模的候选集合。

粗排的目标是：

```text
在较低计算成本下，提高候选 item 的排序质量
```

与召回相比，粗排更强调准确性；与精排相比，粗排更强调效率。

粗排模型通常需要在几千个候选中筛选出几百个 item。它可以使用比召回更多的特征，但不能像精排一样复杂。

粗排和召回的一个重要区别是训练样本不同：

```text
召回：强调不遗漏，负样本常来自全库 item 或 batch 内负样本
粗排：强调接近精排，负样本更常来自曝光未点击样本或召回候选样本
```

因此，即使使用同一种模型结构，例如 DSSM / 双塔模型，它作为召回模型和作为粗排模型时，训练目标、样本构造和线上使用方式也不同。

### 1.3 精排 Ranking

精排阶段对粗排后的候选 item 进行更准确的打分排序。

精排的目标是：

```text
尽可能准确地预测用户对候选 item 的反馈
```

常见优化目标包括：

```text
CTR
CVR
观看时长
互动率
GMV
留存
```

精排模型通常可以使用更丰富的特征：

```text
用户特征
item 特征
上下文特征
用户历史行为特征
交叉特征
序列特征
```

常见精排模型包括：

```text
Wide & Deep
DeepFM
DIN
DIEN
DCN
MMoE
PLE
```

精排是推荐系统效果的核心环节，但由于计算成本较高，通常只能处理较小规模的候选集合。

### 1.4 重排 Re-ranking

重排阶段在精排结果基础上进行最终调整。

重排不只关注单个 item 的分数，还关注整个推荐列表的整体质量。

重排考虑的因素包括：

```text
多样性
新鲜度
去重
业务规则
内容安全
曝光控制
频控
探索与利用平衡
```

精排通常是 point-wise 或 pair-wise 的局部排序，而重排更关注 list-wise 的整体效果。

可以理解为：

```text
精排决定 item 的相关性
重排决定列表的整体体验
```

## 2. 为什么需要多阶段推荐

### 2.1 计算效率

全量 item 数量巨大，无法直接对所有 item 使用复杂模型进行精排。

多阶段架构通过逐步缩小候选规模，使系统能够在有限延迟内完成推荐。

```text
召回：千万级 → 千级
粗排：千级 → 百级
精排：百级 → 十级
重排：生成最终列表
```

### 2.2 不同阶段优化目标不同

推荐系统不是单一模型可以完成的任务。

不同阶段关注点不同：

```text
召回：覆盖潜在兴趣
粗排：快速筛选优质候选
精排：准确预测用户反馈
重排：优化列表整体体验
```

因此，多阶段架构本质上是将复杂推荐问题拆解为多个子问题。

### 2.3 保证多样性与探索空间

如果完全依赖排序模型，系统可能倾向于推荐短期反馈最高的 item，导致热门偏置、兴趣收窄和信息茧房。

召回阶段通过多路召回引入不同来源的候选 item，可以为后续排序保留更多可能性。

因此，召回不仅是为了降低计算量，也承担着控制候选分布、保留探索空间的作用。

## 3. Top-K 推荐 vs CTR 预估

### 3.1 Top-K 推荐

Top-K 推荐关注的是：

```text
从候选 item 集合中选出排名最高的 K 个 item
```

它的目标是排序质量，而不是单个样本的概率校准。

常见形式：

```text
user
→ candidate items
→ scoring / ranking
→ Top-K items
```

常见评估指标：

```text
Recall@K
Precision@K
HitRate@K
NDCG@K
MRR
```

Top-K 推荐更常见于：

```text
召回
序列推荐
离线推荐评估
生成式推荐评估
```

### 3.2 CTR 预估

CTR 预估关注的是：

```text
给定 user、item 和 context，预测用户点击该 item 的概率
```

形式上通常是二分类问题：

```text
P(click = 1 | user, item, context)
```

常见评估指标：

```text
AUC
LogLoss
GAUC
Calibration
```

CTR 预估更常见于：

```text
粗排
精排
广告排序
信息流排序
```

### 3.3 两者关系

Top-K 推荐是推荐任务的输出形式，CTR 预估是排序打分的一种建模方式。

也就是说：

```text
CTR 模型可以用于 Top-K 排序
但 Top-K 推荐不等于 CTR 预估
```

CTR 预估更关注单个 user-item-context 样本的点击概率；Top-K 推荐更关注最终推荐列表中相关 item 是否排在前面。

## 4. 显式反馈 vs 隐式反馈

### 4.1 显式反馈

显式反馈是用户主动表达的偏好。

例如：

```text
评分
点赞
收藏
评价
不感兴趣
```

显式反馈语义清晰，但数据量通常较少。

### 4.2 隐式反馈

隐式反馈是用户行为中隐含的偏好信号。

例如：

```text
点击
浏览
购买
观看时长
加购
停留
分享
```

工业推荐系统中，隐式反馈更加常见。

需要注意：

```text
未点击不一定代表不喜欢
未交互不一定代表负反馈
```

因此，隐式反馈建模通常需要结合：

```text
曝光信息
行为强度
时间衰减
负采样策略
置信度建模
```

## 5. 反馈矩阵与 Embedding 模型

协同过滤的核心数据结构是用户-item 反馈矩阵：

```text
R[user, item]
```

矩阵中的元素表示用户对 item 的反馈强度。

在协同过滤中，模型通过反馈矩阵学习：

```text
user embedding
item embedding
```

并通过相似度或点积预测用户对 item 的偏好：

```text
score(u, i) = user_embedding(u) · item_embedding(i)
```

矩阵分解可以表示为：

```text
R ≈ U · V^T
```

其中：

```text
U：用户 embedding 矩阵
V：item embedding 矩阵
```

协同过滤的本质是：

```text
从用户-item 交互关系中学习用户和 item 的低维表示
```

与基于内容过滤不同，协同过滤主要依赖行为数据，而不是 item 的文本、类目或属性特征。

## 6. 基于内容过滤 vs 协同过滤

### 6.1 基于内容过滤

基于内容过滤根据 item 自身内容特征进行推荐。

内容特征包括：

```text
标题
描述
类目
标签
品牌
图片
文本语义
```

其核心思想是：

```text
用户喜欢过某些内容特征
→ 推荐具有相似内容特征的 item
```

基于内容过滤的表示来源是 item 内容。

### 6.2 协同过滤

协同过滤根据用户群体行为进行推荐。

其核心思想是：

```text
相似用户可能喜欢相似 item
被相似用户共同消费的 item 可能相似
```

协同过滤的表示来源是用户-item 交互关系。

### 6.3 核心区别

```text
基于内容过滤：从内容中构造表示
协同过滤：从行为中学习表示
```

二者都可以得到 user embedding 和 item embedding，但 embedding 的来源不同。

## 7. 序列推荐

序列推荐将用户历史行为看作有序序列。

其核心任务是：

```text
根据用户历史 item 序列，预测下一个 item
```

形式化表示：

```text
[item_1, item_2, ..., item_t]
→ item_{t+1}
```

序列推荐区别于传统协同过滤的一点是，它显式建模行为顺序。

传统协同过滤更关注：

```text
用户和 item 是否发生过交互
```

序列推荐更关注：

```text
用户最近按什么顺序发生了交互
```

常见序列推荐模型包括：

```text
GRU4Rec
SASRec
BERT4Rec
```

## 8. 生成式推荐

生成式推荐将推荐任务转化为生成任务。

传统推荐通常是：

```text
对候选 item 打分
→ 排序
→ 选择 Top-K
```

生成式推荐则是：

```text
根据用户历史
→ 直接生成目标 item 的标识
```

如果直接生成原始 item_id，模型很难学习，因为原始 ID 通常没有语义。

因此，生成式推荐常引入 Semantic ID。

## 9. Semantic ID 初步理解

Semantic ID 是 item 的离散语义标识。

它通常由 item 的文本、内容或语义表示构造得到。

形式上可以表示为：

```text
item
→ semantic representation
→ discrete codes
→ Semantic ID
```

例如：

```text
item_i → <sid_12><sid_98><sid_203>
```

Semantic ID 的作用是将 item 从无语义的原始 ID 转换为更适合生成模型建模的离散 token 序列。

### 9.1 为什么不用原始 item_id

原始 item_id 存在几个问题：

```text
无语义
无法反映 item 之间的相似性
长尾 item 学习困难
不适合语言模型泛化
```

两个语义相近的 item，其原始 ID 可能完全无关。这会导致模型难以利用 item 之间的语义结构。

### 9.2 为什么 Semantic ID 适合 LLM

LLM 擅长建模 token 序列。

如果将 item 表示为 Semantic ID 序列，推荐任务就可以转化为：

```text
用户历史 SID 序列
→ 生成下一个 item 的 SID 序列
```

这样，推荐任务与语言模型的自回归生成形式更加一致。

## 10. MiniOneRec 在哪里

MiniOneRec 属于生成式推荐方向。

它的核心思想是：

```text
将 item 转换为 Semantic ID
再让模型根据用户历史 SID 序列生成下一个 item 的 SID
```

因此，MiniOneRec 不是传统意义上的单纯召回模型，也不是普通 CTR 精排模型。

更准确地说，它结合了：

```text
序列推荐
生成式推荐
Semantic ID 表示学习
LLM / SFT 训练范式
```

MiniOneRec 的推荐流程可以概括为：

```text
item 文本/内容
→ Semantic ID 构建
→ 用户历史 item 序列转换为 SID 序列
→ 模型学习生成下一个 SID
→ SID 映射回真实 item
→ 使用 Recall@K / NDCG@K 评估
```

## 11. 今日核心总结

推荐系统的基本问题是：

```text
在合适的场景下，为用户从大量 item 中选择最合适的一组 item
```

工业推荐系统通常采用多阶段架构：

```text
召回 → 粗排 → 精排 → 重排
```

传统推荐的基本范式是：

```text
user / item embedding → 打分排序
```

序列推荐的基本范式是：

```text
用户历史序列 → 预测下一个 item
```

生成式推荐的基本范式是：

```text
用户历史 → 生成 item token / Semantic ID
```

MiniOneRec 所处的位置是：

```text
以 Semantic ID 为 item 表示的生成式序列推荐方法
```
